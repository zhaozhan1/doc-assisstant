# 政务公文知识库与写作辅助 Skill — 产品规格文档

**版本**: v1.1  
**日期**: 2026-04-30  
**状态**: 待开发

---

## 一、产品背景与目标

### 1.1 背景

用户为政府/行政机关工作人员，长期积累了 500 个以上的工作文档（含 Word、PDF、Excel、PPT、图片截图、压缩包等多种格式），文件内容覆盖通知/公告、方案/规划、汇报类文稿及 PPT，文件分类结构较混杂，时间标记格式不统一。

用户核心痛点：写新公文时需要手动翻找历史文件、难以保持统一文风、起草效率低。

### 1.2 目标

构建一套**本地化、可离线运行**的公文知识库与 AI 写作辅助工具，具备：

1. 自动解析并索引本地文件夹中的历史文档
2. 检索相关历史文件作为参考素材
3. 提取可复用段落和标准表述
4. 学习并模仿语料库的文风与格式规范
5. 根据用户描述直接生成完整公文初稿

---

## 二、核心约束

| 约束项 | 决策 | 原因 |
|--------|------|------|
| 本地文件安全 | **本地文件内容绝不上传云端**，文档解析、向量化、索引全部在本地完成 | 文件可能涉及敏感/涉密内容，优先保密 |
| 在线检索 | **允许**调用云端搜索 API 检索公开在线资料（政策法规、行业标准等），作为补充素材 | 公开信息不涉及泄密风险，可丰富写作素材 |
| LLM 调用 | 优先使用云端 LLM（Claude API / 其他）处理**不含原始文件内容**的 Prompt；本地文件片段注入 Prompt 时，LLM 可以是云端 | Prompt 中的历史文档块属于用户主动使用，由用户自行判断是否可接受 |
| 运行环境 | 本地命令行脚本 + 本地向量数据库 | 文件索引数据留本地，安全可控 |
| 文件规模 | 支持 500 个以上文件，增量更新索引 | 初始语料规模较大 |

> **安全边界说明**：本系统的核心保密原则是"**原始文件不出本地**"。文件内容经向量化后仅存储在本地 ChromaDB 中；在线检索仅发送用户的查询关键词，不携带任何本地文件内容；LLM 生成时注入的历史文档片段由用户知情使用。

## 四、功能模块详细规格

### 4.1 文档解析器（Ingestion Pipeline）

#### 支持格式与处理方式

| 格式 | 解析库 | 提取内容 |
|------|--------|----------|
| `.docx` | `python-docx` | 正文、标题、表格 |
| `.pdf` | `pdfplumber` + OCR fallback（`pytesseract`） | 文本层；扫描件走 OCR |
| `.xlsx` | `openpyxl` | 表格数据转文本描述 |
| `.pptx` | `python-pptx` | 幻灯片文本、标题、备注 |
| `.png/.jpg` | `pytesseract`（OCR） | 图片中的文字内容 |
| `.txt` | 直接读取 | 全文 |

#### 元数据提取规则

- **文件类型自动分类**：使用 LLM 对文档前 500 字进行零样本分类，打标签（通知、公告、请示、报告、方案、规划、会议纪要、合同、其他）
- **时间信息提取**：正则匹配常见格式（`YYYY年MM月DD日`、`YYYY-MM-DD`、`YYYYMMDD`、文件名中的日期），无法提取时记录为"未知"
- **增量更新**：记录每个文件的 `mtime`（修改时间）和 MD5，重建索引时跳过未变更文件

#### 向量化策略

- **分块方式**：按段落切分，每块 400-600 字，相邻块保留 50 字重叠
- **Embedding 模型**：优先使用本地中文模型（`BAAI/bge-large-zh-v1.5`，可通过 Ollama 运行）
- **存储**：ChromaDB（本地持久化），每条记录包含：文本块、文件路径、文件类型标签、时间、块序号

---

### 4.2 写作辅助引擎

#### 模式 A：一步生成模式

**触发**：用户描述写作需求，系统自动完成检索与生成。

**流程**：

```
用户输入（自然语言描述）
    → 意图解析：提取文件类型、主题、关键词
    → 向量检索：Top-K 相关文档块（K=10~15）
    → 文风分析：从检索结果提取格式特征
    → Prompt 构建（见 4.3）
    → LLM 生成完整初稿
    → 输出为 .docx 文件
```

**用户输入示例**：

> "帮我起草一份关于XX项目年度总结汇报的通知，要求各部门12月15日前提交。"

#### 模式 B：分步检索模式

**触发**：用户明确表示"先搜索"或"看看有什么参考"。

**流程**：

```
用户输入检索关键词
    → 向量检索：返回 Top-5 相关文件摘要
    → 展示检索结果列表（文件名、类型、时间、相关片段预览）
    → 用户选择参考文件（可多选）
    → 用户补充写作要求
    → 基于选定素材生成初稿
    → 输出为 .docx 文件
```

---

### 4.3 在线资料检索模块

**定位**：作为本地知识库的补充，检索公开的政策法规、行业标准、上级文件等在线资料，丰富写作素材和政策依据。

**安全边界**：仅向外发送用户输入的查询关键词，**不携带任何本地文件内容**。

#### 检索触发逻辑

| 场景 | 是否触发在线检索 |
|------|----------------|
| 需要引用最新政策法规 | ✅ 自动触发 |
| 需要行业数据或国家标准 | ✅ 自动触发 |
| 纯格式/文风参考 | ❌ 仅使用本地知识库 |
| 用户明确指令"不要联网" | ❌ 强制关闭 |

#### 支持的检索源

- 中国政府网（gov.cn）
- 国家法规数据库（法律法规全文）
- 各部委官方网站公告
- 通用网页搜索（Tavily API / Bing Search API）

#### 检索结果处理

```
查询关键词（由意图解析生成）
    → 调用在线检索 API
    → 获取摘要 + 来源 URL
    → 展示给用户确认（模式 B）/ 直接注入 Prompt（模式 A）
    → 在"素材来源"清单中注明"在线资料：{来源URL}"
```

> **注意**：在线检索结果仅作为写作参考和政策依据补充，不替代用户对内容准确性的最终审核。

---

系统 Prompt 结构（每次生成时动态构建）：

```
[角色设定]
你是一名资深政府文秘，熟悉行政公文写作规范（参照《党政机关公文格式》GB/T 9704-2012）。

[文风参考]
以下是从本地知识库中检索到的相关历史文档片段，请学习其文风、用词习惯和格式结构：
---
{检索到的历史文档块 1}
---
{检索到的历史文档块 2}
---
...（最多 5 个）

[政策依据]（如有在线检索结果）
以下是检索到的相关政策法规摘要，可作为文件依据引用：
---
{在线检索结果摘要 1}（来源：{URL}）
---

[格式要求]
根据文件类型"{文件类型}"，按照以下结构起草：
{对应文件类型的结构模板}

[写作任务]
{用户的具体需求描述}

[输出要求]
- 直接输出完整正文，不要解释
- 使用正式公文语体，避免口语化表达
- 标题使用【】标注层级
- 关键数字、时间、单位保持准确
```

#### 各文件类型结构模板

**通知**：标题 → 主送机关 → 正文（事由+事项+要求） → 落款+日期

**请示/报告**：标题 → 主送机关 → 正文（背景+情况+请示事项/汇报内容） → 结语 → 落款

**方案/规划**：标题 → 前言/背景 → 目标 → 主要任务（分条） → 保障措施 → 附则

**会议纪要**：会议基本信息 → 出席人员 → 议题 → 讨论内容 → 决定事项 → 落款

**汇报 PPT**：封面建议语 → 目录（建议章节） → 各章节要点（分条） → 结语建议语

### 4.4 输出规格

| 项目 | 规格 |
|------|------|
| 输出格式 | `.docx`（主要）、`.md`（可选） |
| 文档样式 | 详见下方字体规范表 |
| 行距 | 28 磅，页边距符合 GB/T 9704-2012 |
| 同步输出 | 附"素材来源"清单，列明引用了哪些历史文件（文件名+时间） |
| 文件命名 | `{文件类型}_{主题关键词}_{生成日期}.docx` |

#### 字体规范表

| 层级 | 字体 | 字号 | 序号格式 | 其他说明 |
|------|------|------|----------|----------|
| 文件标题 | 方正小标宋简体 | 二号 | 无 | 居中 |
| 一级标题 | 方正黑体_GBK | 三号 | 中文数字：一、二、三…… | 顶格 |
| 二级标题 | 方正楷体_GBK | 三号 | 括号中文数字：（一）（二）…… | 顶格 |
| 三级及以下标题 | 方正仿宋_GBK | 三号 | 阿拉伯数字：1. 2. 3. | 顶格 |
| 正文 | 方正仿宋_GBK | 三号 | 无 | 首行空两字 |
| 英文/数字/符号 | Times New Roman | 随所在层级字号 | — | 混排时统一替换 |

> **字体混排说明**：正文及各级标题中出现的英文字母、阿拉伯数字、标点符号（全角除外）统一使用 Times New Roman，中文字符使用对应层级的方正字体，通过 python-docx 的 `run` 级别字体设置实现精确控制。

## 五、Skill 定义（Claude Skill 格式）

本工具同时封装为可在 Claude 中调用的 Skill，供日常对话中直接触发：

```markdown
---
name: gov-doc-writer
description: |
  政务公文写作辅助 Skill。当用户需要起草公文（通知、请示、报告、方案、规划、
  会议纪要、汇报PPT等）时触发。会自动检索本地知识库中的历史文件作为参考，
  模仿其文风和格式，生成符合《党政机关公文格式》规范的完整初稿。
  支持一步生成（描述需求→直接出稿）和分步检索（先看参考→再出稿）两种模式。
  所有文件处理均在本地完成，不上传至云端。
---
```

**触发关键词示例**：起草、拟稿、写一个通知、帮我写报告、出一份方案、整理会议纪要

## 六、技术栈选型

| 组件 | 选型 | 备注 |
|------|------|------|
| 运行环境 | Python 3.10+ | 跨平台 |
| 文档解析 | python-docx, pdfplumber, python-pptx, openpyxl, pytesseract | 全部开源，本地运行 |
| OCR | Tesseract 5.x + 中文语言包 | 本地运行 |
| Embedding | BAAI/bge-large-zh-v1.5（via Ollama） | 本地运行，中文效果最优 |
| 向量数据库 | ChromaDB（本地持久化） | 轻量、无需服务器，本地存储 |
| LLM | Claude API（默认）/ Ollama 本地模型（可选） | 云端 LLM 不接触原始文件；Prompt 中的文档片段由用户知情使用 |
| 在线检索 | Tavily API / Bing Search API | 仅发送查询关键词，检索公开资料 |
| 输出生成 | python-docx（Word格式化输出） | 本地生成 |
| 接口形式 | 命令行（CLI）+ 可选 Web UI（Gradio） | 后期扩展 |

## 八、配置文件规格（config.yaml）

```yaml
knowledge_base:
  source_folder: "/path/to/your/documents"   # 源文件夹路径
  db_path: "./chroma_db"                      # 向量数据库存储路径（本地）
  chunk_size: 500                             # 分块字数
  chunk_overlap: 50                           # 重叠字数

embedding:
  model: "bge-large-zh-v1.5"                 # Embedding 模型（本地运行）
  provider: "ollama"                          # ollama（本地）

llm:
  provider: "anthropic"                       # anthropic | ollama
  model: "claude-sonnet-4-20250514"           # 云端模型（不接触原始文件）
  api_key: ""                                 # Claude API Key

online_search:
  enabled: true                               # 是否启用在线检索
  provider: "tavily"                          # tavily | bing
  api_key: ""                                 # 在线检索 API Key
  sources:                                    # 优先检索域名
    - "gov.cn"
    - "npc.gov.cn"
    - "moj.gov.cn"
  max_results: 3                              # 每次最多返回条数

output:
  format: "docx"                              # docx | md
  save_path: "./output"
  include_sources: true                       # 是否附素材来源清单（含在线来源）
```

## 十、已知限制与风险

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| OCR 质量不稳定 | 低质量图片截图识别率较低 | 对置信度低的文件标记警告，人工复核 |
| 文件分类误差 | 混杂文件自动分类可能出错 | 提供手动修正标签的命令 |
| 在线检索结果质量 | 搜索结果可能不准确或过时 | 在素材清单中标注来源，用户自行核实后使用 |
| LLM 云端调用 | Prompt 中含有历史文档片段，存在理论上的数据外流 | 明确告知用户；提供 Ollama 本地 LLM 选项供高保密场景使用 |
| 首次建库耗时 | 500+ 文件首次索引预计 30-60 分钟 | 显示进度条，支持断点续传 |
| 涉密文件管控 | 无法自动识别文件密级 | 文档说明：用户需自行确认源文件夹无涉密文件后再运行 |
