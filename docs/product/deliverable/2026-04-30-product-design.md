# 公文助手（Doc-Assistant）— 整体产品设计文档

**版本**: v1.0
**日期**: 2026-04-30
**状态**: 已确认

---

## 一、系统定位与核心边界

### 1.1 系统名称

**公文助手（Doc-Assistant）** — 本地化政务公文知识库与 AI 写作辅助系统。

### 1.2 一句话定位

从用户积累的历史文档中自动构建知识库，检索相关素材，辅助生成符合公文规范的完整初稿。

### 1.3 核心安全边界

| 原则 | 说明 |
|------|------|
| 原始文件不出本地 | 文档解析、向量化、索引全部本地完成，文件内容不离开本机 |
| 用户知情使用 LLM | 主动检索到的文档片段注入 Prompt 时，由用户判断是否可接受 |
| 在线检索仅发关键词 | 调用外部搜索 API 时不携带任何本地文件内容 |

### 1.4 目标用户与场景

- **用户**：政府/行政机关工作人员（单用户）
- **核心场景**：写新公文时，自动从历史文档中找到参考素材，学习文风，生成初稿
- **文档规模**：初期 500+，架构预留 5000 扩展能力

### 1.5 第一版成功标准（生产可用）

1. 能导入并解析 7 种格式的历史文档：docx / pdf / xlsx / pptx / png+jpg（OCR） / txt / 压缩包（自动解压后解析内部文件）
2. 支持的压缩包格式：`.zip` / `.rar` / `.7z`，自动递归解压（嵌套压缩包也处理），提取内部可识别格式的文件进入解析流水线
3. 文档自动分类准确率 ≥ 85%（可手动修正）
4. 根据自然语言描述生成格式正确的公文初稿，文风接近历史文件
5. 输出符合 GB/T 9704-2012 排版标准的 .docx 文件
6. 支持在线政策法规检索作为补充素材
7. Web UI 可正常操作全流程

## 二、系统架构

### 2.1 总体架构

采用 **单体 Web 应用** 架构，分为三层：

```
┌─────────────────────────────────────────────┐
│              Web 前端（浏览器）               │
│   文档管理 / 检索 / 写作辅助 / 系统设置      │
└──────────────────┬──────────────────────────┘
                   │ HTTP / WebSocket
┌──────────────────▼──────────────────────────┐
│            Web 后端（FastAPI）                │
│                                             │
│  ┌─────────┐ ┌─────────┐ ┌──────────────┐  │
│  │文档解析  │ │检索引擎  │ │写作辅助引擎  │  │
│  │模块      │ │模块      │ │模块          │  │
│  └────┬────┘ └────┬────┘ └──────┬───────┘  │
│       │           │             │           │
│  ┌────▼───────────▼─────────────▼───────┐   │
│  │         本地向量数据库（持久化）       │   │
│  └──────────────────────────────────────┘   │
└─────────────────────────────────────────────┘
                   │
           ┌───────┴────────┐
           │ 外部 API 调用   │
           │ LLM / 在线检索  │
           └────────────────┘
```

### 2.2 后端模块划分（4 个核心模块）

| 模块 | 职责 | 对外依赖 |
|------|------|----------|
| **文档解析模块 (Ingestion)** | 文件导入、解压、格式识别、文本提取、分类、分块、向量化入库 | 本地 Embedding 模型 |
| **检索模块 (Retrieval)** | 本地向量检索 + 在线资料检索 + 结果融合排序 | 向量数据库 / 在线搜索 API |
| **写作辅助模块 (Generation)** | 意图解析、Prompt 构建、LLM 调用、公文生成、Word 格式化输出、Word 转 PPT | LLM API |
| **系统管理模块 (System)** | 配置管理、知识库状态监控、文件管理、模板管理 | 无 |

### 2.3 数据流

**导入流程**：

```
用户选择文件/文件夹
  → 压缩包自动解压（如有）
  → 格式识别 & 文本提取
  → 自动分类（LLM 零样本分类）
  → 段落分块 & 向量化
  → 存入本地向量数据库
```

**写作流程（模式 A — 一步生成）**：

```
用户描述写作需求
  → 意图解析（提取文件类型、主题、关键词）
  → 并行检索：本地知识库 + 在线资料（可选）
  → Prompt 构建（角色 + 文风参考 + 政策依据 + 模板 + 任务）
  → LLM 生成初稿
  → Word 格式化输出
```

**写作流程（模式 B — 分步检索）**：

```
用户输入检索关键词
  → 返回相关文件列表（文件名、类型、时间、片段预览）
  → 用户选择参考文件 + 补充写作要求
  → 基于选定素材生成初稿
  → Word 格式化输出
```

**写作流程（模式 C — Word 转 PPT）**：

```
用户上传/选择已有 Word 文档
  → 解析文档结构（提取标题层级、段落要点）
  → LLM 总结各章节为 PPT 页面内容（标题 + 要点）
  → 生成 .pptx 文件（含封面、目录、各章节页、结语页）
```

### 2.4 LLM 调用策略

**原则：本地 LLM 为主，云端 LLM 为可选补充。**

```
┌─────────────────────────────────────┐
│         LLM API 抽象层              │
│  统一接口：chat / classify / embed   │
├──────────┬──────────┬───────────────┤
│  Ollama  │  Claude  │  可扩展其他   │
│  (默认)   │  (可选)   │  (预留)      │
│  本地运行 │  云端调用 │              │
└──────────┴──────────┴───────────────┘
```

| 设计要点 | 说明 |
|----------|------|
| **统一抽象接口** | 所有 LLM 调用（分类、生成、Embedding）通过统一的 Provider 接口，上层模块不感知具体后端 |
| **默认本地优先** | 分类、文风分析、Embedding 等高频调用走本地 Ollama，避免频繁网络开销 |
| **云端降级可选** | 用户可配置允许云端 LLM 处理复杂生成任务，但默认关闭 |
| **配置化切换** | config.yaml 中配置 provider、base_url 和 model，运行时可切换，无需改代码 |

## 三、文档解析模块 (Ingestion)

### 3.1 功能边界

负责将用户提供的文件/文件夹转化为可检索的结构化数据，存入本地向量数据库。

### 3.2 支持的输入格式

| 格式 | 提取内容 | 备注 |
|------|----------|------|
| `.docx` | 正文、标题、表格 | — |
| `.pdf` | 文本层；扫描件走 OCR | — |
| `.xlsx` | 表格数据转文本描述 | — |
| `.pptx` | 幻灯片文本、标题、备注 | — |
| `.png/.jpg` | OCR 提取文字 | — |
| `.txt` | 全文 | — |
| `.zip/.rar/.7z` | 自动解压，递归提取内部可识别格式文件 | 嵌套压缩包也处理 |

### 3.3 处理流水线

```
输入：文件/文件夹路径
  │
  ├─ 1. 预处理：压缩包自动解压 → 文件格式识别 → 过滤不支持的格式
  │
  ├─ 2. 文本提取：按格式调用对应解析器 → 提取纯文本 + 保留结构信息（标题层级）
  │
  ├─ 3. 元数据提取：
  │     ├─ 文件名、文件路径、导入时间
  │     ├─ 文档内时间信息（正则匹配多种日期格式）
  │     └─ 文件类型自动分类（LLM 零样本分类，取前 500 字）
  │
  ├─ 4. 文本分块：按段落切分，每块 400-600 字，相邻块 50 字重叠
  │
  ├─ 5. 向量化：通过本地 Embedding 模型生成向量
  │
  └─ 6. 入库：文本块 + 向量 + 元数据存入本地向量数据库
```

### 3.4 增量更新机制

- 记录每个文件的 MD5 + 修改时间
- 重复导入时跳过未变更文件
- 文件内容变更时删除旧数据，重新处理入库

### 3.5 异步工作流

- 文档导入为长时间操作（500+ 文件），采用异步任务队列模式
- 每个文件的解析、分块、向量化作为独立异步任务
- Web 前端通过 WebSocket 实时推送进度（当前文件、总数、成功/失败/跳过计数）
- 支持中途取消导入，已处理的数据保留

```
导入任务状态机：

  Pending → Running → Completed
                ├→ Failed（记录失败原因，继续处理下一个）
                ├→ Cancelled（用户主动取消）
                └→ 每个文件独立状态，互不影响
```

### 3.6 平台兼容性

- 支持 macOS 系统
- 文件路径处理统一使用 `pathlib.Path`
- 压缩包解压统一使用 Python 库处理（`zipfile` / `py7zr`），不依赖系统命令行工具
- OCR 等外部工具路径通过配置文件指定

### 3.7 关键 API 接口

```
ingest.import_path(path)          # 导入文件/文件夹
ingest.get_status()               # 获取导入进度/状态
ingest.get_progress(task_id)      # 获取导入任务进度（WebSocket 推送）
ingest.cancel(task_id)            # 取消正在进行的导入任务
ingest.reindex(file_id)           # 单文件重新索引
ingest.delete(file_id)            # 删除文件及其索引数据
ingest.list_files(filters)        # 列出已导入文件（支持按类型/时间筛选）
ingest.update_classification()    # 手动修正文件分类标签
```

## 四、检索模块 (Retrieval)

### 4.1 功能边界

根据用户查询，从本地知识库和在线资料源中检索相关内容，返回融合排序后的结果。

### 4.2 双路检索架构

```
用户查询
  │
  ├─ 路径 A：本地知识库检索
  │   └─ 查询向量化 → 向量数据库相似度搜索 → Top-K 文档块
  │
  ├─ 路径 B：在线资料检索（可选，用户可关闭）
  │   └─ 提取关键词 → 调用搜索 API → 获取摘要 + 来源 URL
  │
  └─ 结果融合排序 → 返回统一结果列表
```

### 4.3 本地检索

| 要素 | 说明 |
|------|------|
| 查询方式 | 用户输入自然语言 → 向量化 → 余弦相似度检索 |
| 返回字段 | 文件名、文件类型、时间、相关片段预览、相似度分数 |
| 默认 Top-K | 模式 A 取 10-15 条，模式 B 取 5 条 |
| 过滤 | 支持按文件类型、时间范围筛选 |

### 4.4 在线检索

| 要素 | 说明 |
|------|------|
| 触发逻辑 | 需要政策法规/行业数据时自动触发；纯文风参考不触发；用户可强制关闭 |
| 检索源 | gov.cn / npc.gov.cn / moj.gov.cn / 通用网页搜索 |
| 安全边界 | 仅发送查询关键词，不携带本地文件内容 |
| 结果处理 | 返回摘要 + 来源 URL，用户可确认后使用 |

### 4.5 结果融合

- 本地和在线结果统一格式
- 按相关度分数排序，本地结果优先
- 标注来源类型（本地文件 / 在线资料）

### 4.6 关键 API 接口

```
retrieval.search(query, top_k, filters)        # 综合检索（本地 + 在线）
retrieval.search_local(query, top_k, filters)   # 仅本地检索
retrieval.search_online(query, max_results)     # 仅在线检索
```

## 五、写作辅助模块 (Generation)

### 5.1 功能边界

根据用户需求和检索到的素材，通过 LLM 生成符合公文规范的完整初稿，并格式化输出为 Word 或 PPT 文件。

### 5.2 三种写作模式

**模式 A — 一步生成**：

```
用户输入自然语言描述
  → 意图解析：提取文件类型、主题、关键词
  → 自动检索（本地 + 在线）
  → Prompt 构建
  → LLM 生成初稿
  → Word 格式化输出
```

**模式 B — 分步检索**：

```
用户输入检索关键词
  → 展示检索结果列表
  → 用户选择参考文件 + 补充写作要求
  → Prompt 构建
  → LLM 生成初稿
  → Word 格式化输出
```

**模式 C — Word 转 PPT**：

```
用户上传/选择已有 Word 文档
  → 解析文档结构（提取标题层级、段落要点）
  → LLM 总结各章节为 PPT 页面内容（标题 + 要点）
  → 生成 .pptx 文件（含封面、目录、各章节页、结语页）
```

### 5.3 Prompt 构建策略

动态组装 5 个部分：

| 部分 | 内容 | 来源 |
|------|------|------|
| 角色设定 | 资深政府文秘，熟悉 GB/T 9704-2012 | 固定模板 |
| 文风参考 | Top-5 相关历史文档片段 | 本地检索 |
| 政策依据 | 在线检索结果摘要 + 来源 URL（如有） | 在线检索 |
| 格式要求 | 对应文件类型的结构模板 | 模板库 |
| 写作任务 | 用户的具体需求描述 | 用户输入 |

### 5.4 模板管理

- 内置模板：通知、公告、请示、报告、方案、规划、会议纪要、合同/协议、工作总结、领导讲话稿、调研报告、汇报 PPT
- 每个模板定义：结构大纲 + 各段写作要点 + 格式规范
- 用户可在 Web UI 中自定义新模板或修改内置模板
- 模板存储为 JSON/YAML 格式，方便编辑

### 5.5 输出规格

| 项目 | 规格 |
|------|------|
| 主输出 | `.docx` 文件，符合 GB/T 9704-2012 排版标准 |
| 可选输出 | `.md` 格式 |
| 附件 | 素材来源清单（引用的历史文件名 + 时间 + 在线来源 URL） |
| 文件命名 | `{文件类型}_{主题关键词}_{生成日期}.docx` |

### 5.6 字体规范

| 层级 | 字体 | 字号 | 格式 |
|------|------|------|------|
| 文件标题 | 方正小标宋简体 | 二号 | 居中 |
| 一级标题 | 方正黑体_GBK | 三号 | 首行空两字，中文数字序号 |
| 二级标题 | 方正楷体_GBK | 三号 | 首行空两字，括号中文数字 |
| 三级标题 | 方正仿宋_GBK | 三号 | 首行空两字，阿拉伯数字 |
| 正文 | 方正仿宋_GBK | 三号 | 首行空两字 |
| 英文/数字 | Times New Roman | 随层级 | run 级别字体控制 |

行距：28 磅，页边距符合 GB/T 9704-2012。

### 5.7 关键 API 接口

```
generation.generate_one_step(description)                  # 模式 A
generation.search_and_preview(query)                        # 模式 B - 检索预览
generation.generate_with_selection(selection, requirements)  # 模式 B - 生成
generation.generate_pptx(docx_path)                         # 模式 C - Word 转 PPT
generation.list_templates()                                 # 列出可用模板
generation.get_template(template_id)                        # 获取模板详情
generation.create_template(template_data)                   # 创建自定义模板
generation.update_template(template_id, template_data)      # 修改模板
```

## 六、Web 前端设计

### 6.1 功能边界

为所有后端功能提供浏览器操作界面，单页应用，本地运行。

### 6.2 页面结构

```
┌─────────────────────────────────────────────┐
│  顶部导航栏：公文助手 Doc-Assistant          │
│  [知识库]  [写作]  [模板管理]  [设置]        │
├─────────────────────────────────────────────┤
│                                             │
│              各页面内容区                     │
│                                             │
└─────────────────────────────────────────────┘
```

### 6.3 四个核心页面

**1. 知识库页面**

- 导入区域：选择文件/文件夹上传，支持拖拽
- 导入进度：实时显示（WebSocket），当前文件/总数/成功/失败/跳过
- 文件列表：已导入文件表格（文件名、类型标签、时间、状态），支持按类型/时间筛选
- 操作：重新索引、删除、手动修正分类标签
- 知识库统计：文件总数、各类型分布、最后更新时间

**2. 写作页面**

- 模式切换：一步生成 / 分步检索 / Word 转 PPT
- 一步生成：输入框（自然语言描述）→ 生成按钮 → 结果预览区 → 下载 .docx
- 分步检索：搜索框 → 结果列表（可勾选）→ 补充要求输入框 → 生成按钮 → 结果预览 → 下载
- Word 转 PPT：上传/选择 Word 文件 → 生成按钮 → PPT 预览 → 下载 .pptx
- 生成过程：显示 LLM 生成状态，流式输出文本预览
- 生成完成后附带素材来源清单

**3. 模板管理页面**

- 模板列表：内置模板 + 用户自定义模板
- 操作：查看模板详情、新建模板、编辑模板、删除自定义模板
- 模板编辑器：结构大纲编辑（标题层级 + 各段写作要点）

**4. 设置页面**

- 知识库配置：源文件夹路径、数据库路径、分块参数
- LLM 配置：Provider（Ollama/Claude/其他）、Base URL、模型选择、API Key
- 在线检索配置：开关、Provider、API Key、优先检索域名
- 输出配置：默认格式、保存路径、是否附带素材来源

## 八、开发阶段拆分

### 8.1 阶段总览

| 阶段 | 名称 | 功能范围 | 交付物 |
|------|------|----------|--------|
| **一** | 项目基础 + 文档解析 | 项目初始化、配置管理、多格式文档解析（含压缩包）、文本分块、向量化入库、增量更新 | 可运行的导入命令 + 单元测试 |
| **二** | 检索引擎 + 知识库管理 | 本地向量检索、在线检索集成、结果融合排序、知识库文件管理（列表/筛选/删除/修正标签） | 可运行的检索命令 + 单元测试 |
| **三** | 写作辅助核心 | 意图解析、Prompt 构建、模板管理（内置+自定义）、LLM 生成、Word 格式化输出、Word 转 PPT | 可运行的生成命令 + 单元测试 |
| **四** | Web UI | 4 个核心页面（知识库/写作/模板管理/设置）、异步任务进度推送、流式生成预览、文件下载 | 完整可用的 Web 应用 |
| **五** | 优化与加固 | 导入性能优化（并行处理）、检索准确率调优、错误处理完善、日志系统、5000 文档扩展验证 | 生产可用的完整系统 |

### 8.2 阶段间依赖关系

```
阶段一（文档解析）→ 阶段二（检索引擎）→ 阶段三（写作辅助）
                                              ↓
                                         阶段四（Web UI）
                                              ↓
                                         阶段五（优化）
```

阶段一、二、三串行依赖；阶段四依赖前三阶段的 CLI 功能；阶段五在完整功能基础上优化。

## 九、已知限制与风险

| 风险 | 说明 | 缓解措施 |
|------|------|----------|
| OCR 质量不稳定 | 低质量图片截图识别率较低 | 对置信度低的文件标记警告，人工复核 |
| 文件分类误差 | 混杂文件自动分类可能出错 | 提供手动修正标签的命令 |
| 在线检索结果质量 | 搜索结果可能不准确或过时 | 在素材清单中标注来源，用户自行核实 |
| 首次建库耗时 | 500+ 文件首次索引预计 30-60 分钟 | 异步任务 + 进度条，支持断点续传 |
| 涉密文件管控 | 无法自动识别文件密级 | 文档说明：用户需自行确认源文件夹无涉密文件后再运行 |
| 本地 LLM 性能 | 本地模型生成质量可能不如云端大模型 | 提供 Base URL 配置，方便切换更强模型 |
